# planning/ — working plans and in-flight work

This folder holds **working** planning material: feature plans, refactor investigations, and their status. It is distinct from `context/` (stable canon) and `docs/` (the public site).

## Source-of-truth rules

1. **Plans are provisional; `context/` is canonical.** When a plan is accepted and changes how the library works, fold the outcome into the relevant `context/` doc. If it closes off an alternative, add a `context/decisions/` record.
2. **`STATUS.md` is the index.** Every active feature/refactor folder gets a row. Start there.
3. **One folder per unit of work**, lowercase kebab-case (`multiple-ventilation-systems`).

## Layout

```
planning/
  README.md          # read order
  STATUS.md          # master index of active work
  features/          # new-capability plans      (features/<slug>/)
  refactor/          # cross-cutting refactors    (refactor/<slug>.md or <slug>/)
  archive/           # completed/superseded work  (archive/<slug>/ + README index)
```

## Feature/refactor folder contract

A rich unit of work uses:

```
<slug>/
  README.md      # router: scope, read order
  PRD.md         # what/why (behavior contract)
  STATUS.md      # current state, next step, blockers
  PLAN.md        # optional implementation sequence
```

For a small or dormant item, a single `<slug>.md` (like `refactor/multiple-ventilation-systems.md`) is enough.

## Archiving completed work

When an item is `Complete` or `Superseded`: fold its outcome into `context/` (and add a `context/decisions/` record if it settled a choice), then **move** its folder to `planning/archive/<slug>/` unchanged and add a row to `planning/archive/README.md`. Archive is flat by slug (findable by name), never nested by date — see that README.

## Status vocabulary

`Requested` · `Scoped` · `In progress` · `Blocked` · `Implemented` · `Complete` · `Deferred` · `Superseded`.
